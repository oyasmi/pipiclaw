# 并发与容量参考（Scaling and Concurrency Reference）

这份文档帮助你在部署 Pipiclaw 前评估它能服务多少用户，以及在多人同时使用时需要注意什么。

如果你还没有看过部署文档，建议先看：

- [deployment-and-operations.md](./deployment-and-operations.md)
- [configuration.md](./configuration.md)

## 会话隔离模型（Session Isolation Model）

Pipiclaw 按**会话通道（channel）**隔离状态。每个通道有独立的对话历史、记忆和执行上下文。

通道 ID 的生成规则：

| 场景 | 通道 ID 格式 | 说明 |
|------|-------------|------|
| 私聊 | `dm_{staffId}` | 按发送者隔离 |
| 群聊 | `group_{conversationId}` | 按群隔离 |

这意味着：

- 你和同事分别私聊同一个 Pipiclaw 实例，各自是独立通道
- 同一个人在两个不同群里和 Pipiclaw 交互，也是两个独立通道
- 不同通道之间的对话历史、`SESSION.md`、`MEMORY.md` 互不可见

## 并发处理方式（Concurrency Model）

### 不同通道之间：并行

不同通道的消息处理是完全并行的。通道 A 正在执行工具调用，不会阻塞通道 B 接收和处理新消息。

Pipiclaw 是单进程 Node.js 应用，并行能力来自 async/await 异步模型：LLM 请求和网络 I/O 等待期间不占用 CPU，其他通道可以正常推进。

### 同一通道内：串行

同一个通道同一时刻只运行一个任务。当通道正在处理时，新消息的行为是：

| 新消息类型 | 行为 |
|-----------|------|
| 普通消息 | 默认作为 steer（即时干预），在当前工具步骤完成后插入 |
| `/followup <消息>` | 排队到当前任务完成后再处理 |
| `/stop` | 中止当前任务 |
| `/help` | 立即响应，不影响当前任务 |
| 其他 `/` 命令 | 拒绝执行，提示当前正忙 |

## 每通道资源开销（Per-Channel Resource Footprint）

每个通道创建后，会在内存中常驻以下对象：

| 组件 | 典型大小 | 说明 |
|------|---------|------|
| Agent + AgentSession | 2-5 MB | LLM 交互状态和消息缓存 |
| SessionManager | 0.5-5 MB | 对话历史管理，随对话增长 |
| MemoryLifecycle | 0.3-1 MB | 记忆整理和持久化状态 |
| 工具元数据 | 0.2-0.5 MB | 注册的工具定义 |
| **单通道合计** | **约 4-16 MB** | 取决于对话长度和复杂度 |

除了常驻内存，运行时还有临时开销：

- **工具执行**：每次 bash 调用会 spawn 一个子进程，执行完即销毁
- **记忆维护**：每隔几轮对话会触发一次 sidecar LLM 调用，用于更新 `SESSION.md` 和整理记忆
- **上下文压缩**：对话接近上下文窗口上限时，会触发压缩（compaction），需要一次 LLM 调用

磁盘方面，每个通道在 `workspace/` 下有独立目录，存放 `context.jsonl`、`SESSION.md`、`MEMORY.md`、`HISTORY.md` 等文件。长期运行后，单通道磁盘占用通常在 100 KB 到几 MB 之间。

## 已知瓶颈（Known Bottlenecks）

### 1. 通道 Runner 不会自动释放

通道的 Runner 对象创建后会一直留在内存中，直到进程重启。没有空闲驱逐机制。

这意味着：如果累计有大量不同通道和 Pipiclaw 交互过，即使它们已经很久不活跃，内存仍然被占用。

**影响**：长期运行的实例，通道数量持续增长时，内存会逐渐上升。

**缓解方式**：定期重启进程可以释放所有通道状态。重启后通道的持久化记忆（`MEMORY.md`、`HISTORY.md`）不受影响。

### 2. LLM API 是主要约束

每个通道在活跃对话时会发起 LLM API 请求。此外，记忆维护和上下文压缩也会产生额外的 LLM 调用。一个活跃通道的一轮对话，平均可能产生 2-5 次 LLM API 请求。

当前 Pipiclaw 没有全局 LLM API 限流。如果多个通道同时活跃，可能触发模型提供方的速率限制（rate limit），导致部分请求失败或延迟。

### 3. 单 WebSocket 连接

所有通道共享一个钉钉 WebSocket 连接。正常使用下这不是性能瓶颈，但它是单点：如果连接断开，所有通道的消息收发都会暂停，直到重连完成。

当前运行时会自己接管重连，而不是依赖 SDK 的自动重连。每次重连前会先清理旧 socket，并在必要时对长时间无响应的连接做强制终止，再按退避策略重试。这样可以降低复杂网络环境下僵尸连接和重复重连叠加的风险。

### 4. 子进程无池化

工具执行（bash 命令）每次都 spawn 新进程，没有进程池。多个通道同时执行工具时，会同时存在多个子进程。通常这不是问题，但在极端并发下可能消耗较多内存和文件描述符。

## 容量估算参考（Capacity Estimation）

下面以一台 4C8G Linux 服务器为例，给出粗略的容量参考。

### 内存预算

```text
总内存:                    8 GB
- 系统 + Node.js 基础:    ~0.5 GB
- 钉钉 Bot + 全局状态:    ~0.2 GB
- 可分配给通道:           ~7.3 GB
```

### 按使用模式估算

| 使用模式 | 每通道内存 | 可支撑通道数 | 典型场景 |
|---------|-----------|------------|---------|
| 轻度 | 4-6 MB | 150-180 | 偶尔问答，对话较短 |
| 中度 | 8-12 MB | 60-90 | 日常开发协助，正常对话长度 |
| 重度 | 16-30 MB | 25-45 | 长对话、频繁工具调用、使用子代理 |

说明：

- **通道数**指的是累计创建过的通道总数（包括不活跃的），因为 Runner 不会自动释放
- 如果定期重启进程，只需要关注同时存在的活跃通道数
- 上面的数字是内存维度的理论上限，实际瓶颈通常先出现在 LLM API 侧

### 真正的并发瓶颈：LLM API

内存通常不是第一个遇到的限制。更常见的约束是：

| 约束 | 说明 |
|------|------|
| API 速率限制 | 模型提供方通常有 RPM（每分钟请求数）限制；10-30 个通道同时活跃时可能触发 |
| API 延迟叠加 | 每个通道的 LLM 等待是异步的，不会互相阻塞，但模型提供方的总吞吐有限 |
| 记忆维护开销 | 每个活跃通道额外产生 sidecar LLM 调用，会消耗 API 配额 |

粗略地说，在一台 4C8G 服务器上：

- **同时注册的通道数**（含不活跃的）：100-150 个
- **同时活跃对话的通道数**：10-30 个，取决于模型提供方的速率限制

### 其他硬件维度

| 资源 | 是否瓶颈 | 说明 |
|------|---------|------|
| CPU | 通常不是 | Node.js 事件循环中 LLM 等待是异步的，4 核足够 |
| 磁盘 I/O | 通常不是 | 每通道的 JSONL 和 MD 写入量不大，SSD 下无压力 |
| 网络 | 看模型部署 | 如果模型在远端，带宽通常不是问题；如果走代理，注意代理的连接数限制 |

## 部署建议（Deployment Recommendations）

### 小团队（10 人以下）

一个 Pipiclaw 实例通常足够。建议：

- 2C4G 以上的机器
- 用 `allowFrom` 控制使用范围
- 关注模型提供方的 API 配额是否充足

### 中等规模（10-50 人）

仍然可以用单实例，但需要更多关注：

- 建议 4C8G 以上
- 监控 Node.js 进程的内存使用（RSS）
- 根据使用量评估是否需要更高的 API 速率配额
- 考虑定期重启以释放不活跃通道的内存

### 更大规模

如果预期同时活跃的通道数经常超过 30 个，建议：

- 部署多个 Pipiclaw 实例，按群或团队分配
- 每个实例绑定不同的钉钉应用
- 各实例独立管理自己的工作区和配置

## 监控建议（Monitoring Recommendations）

长期运行时，建议关注以下指标：

| 指标 | 获取方式 | 关注阈值 |
|------|---------|---------|
| Node.js RSS 内存 | `ps aux` 或进程管理器 | 接近可用内存的 80% |
| 进程日志中的 API 报错 | `journalctl` 或 `pm2 logs` | rate limit 或 timeout 错误频繁出现 |
| 钉钉 WebSocket 重连 | 进程日志中的 reconnect / forcing socket termination 记录 | 频繁重连通常说明网络不稳定；如果反复出现 forced termination，优先排查内网代理、防火墙或长连接质量 |
| 磁盘使用 | `du -sh ~/.pi/pipiclaw/workspace/` | 长期运行后关注是否需要清理旧通道目录 |
